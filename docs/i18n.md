# Internationalization (i18n)

The workspace ships with shared English and Russian translations for frontend copy and backend problem responses. The goal is to keep user-facing text localizable while preserving stable error codes for clients, logs, and alerting.

## Locale catalog placement

Locale JSON catalogs live at the repository root as thin scoped files under `i18n/<locale>/<scope>/<component>.json`, currently `i18n/en/**` and `i18n/ru/**`. Keep catalog assets outside compiled libraries, keep supported locales constrained to `en` and `ru`, and keep `en` as the fallback locale. Do not create unrelated package-root locale directories.

`libs/common/i18n` remains the shared TypeScript runtime for these catalogs: it imports the root JSON assets, merges each locale's thin files into the same flat dotted key map, exposes supported locale types, translates keys, interpolates params, resolves request locale, and provides request middleware. Keep catalog data in root `i18n/`; keep translation code and tests in `libs/common/i18n`; keep backend-only Nest/nestjs-i18n integration in `libs/backend/common/intl`.

Catalog scopes are ownership boundaries:

- `i18n/<locale>/common/shared.json` for shared UI, runtime overlay, language, theme, and common status copy.
- `i18n/<locale>/common/errors.json` for shared API/problem/validation fallback copy.
- `i18n/<locale>/landing/app.json` for the landing frontend.
- `i18n/<locale>/admin/*.json` for admin frontend shell and pages.
- `i18n/<locale>/user/*.json` for user frontend shell, auth, social auth, and TMA copy.
- `i18n/<locale>/bots/*.json` for Telegram bot and Discord interaction copy.

Frontend apps must pass scoped catalog sets into `FrontendI18nProvider`: landing imports the landing catalog set, user imports the user catalog set, and admin imports the admin catalog set. Shared UI and API-support fallback copy use only the common shared/error catalog set by default. Do not import the full `@app/common/i18n` catalog from frontend code, because that export intentionally includes bot and Discord interaction copy for backend/runtime consumers.

## Locale detection and persistence order

Backend request locale resolution uses the first supported locale found in this order:

1. `?lang=` or `?locale=` query parameter
2. `x-locale` or `x-language` header
3. `locale` or `lang` cookie
4. `Accept-Language` header
5. Default fallback locale (`en`)

Authenticated frontend sessions additionally persist profile preferences in `auth_users.locale` and `auth_users.theme`. Locale remains constrained to supported values (`en`/`ru`) when present. Theme is constrained to `system`, `light`, or `dark`, and defaults to `system` for existing and new users. After a bearer token is available, the user and admin frontends load `/auth/me` and/or profile data, apply saved locale/theme first, then continue using browser-safe fallback behavior if no saved preference exists:

1. saved authenticated user locale from `/auth/me` or profile data
2. `?lang=` or `?locale=` query parameter
3. `boilerplate.locale` in `localStorage`
4. `locale` or `lang` cookie
5. `navigator.languages` / `navigator.language`
6. Default fallback locale (`en`)

Guests use only browser-safe fallback behavior from URL/localStorage/cookie/browser/default values.

The frontend language switcher always persists locale locally in `localStorage` and a `locale` cookie. The theme switcher persists `boilerplate.theme` in `localStorage`, sets `document.documentElement.dataset.themePreference` to `system|light|dark`, and sets `document.documentElement.dataset.theme` to the resolved `light|dark` value. System-theme preference tracks `prefers-color-scheme` changes and applies updates live.

For authenticated users, language/theme switches call the generated `authApi.authControllerUpdatePreferences` wrapper for `PATCH /auth/me/preferences` so both preferences follow future sessions. `PATCH /auth/me/locale` remains as a compatibility wrapper that routes through the same preference update path. User and admin frontend API calls must always send `Accept-Language` with the current resolved locale; backend APIs still localize each request from that frontend-provided header (or the backend fallback chain above).

## Auth preferences API

The auth preference contract is intentionally partial-update friendly:

```json
{
  "locale": "en",
  "theme": "system"
}
```

Both fields are optional on `PATCH /auth/me/preferences`. Omitted values are preserved, unsupported locale or theme values return a localized 400 problem response, and successful updates return the updated authenticated user view. `GET /auth/me`, auth session DTOs, and frontend generated auth types include `theme` alongside `locale` so app/admin shells can hydrate preferences without a second preferences-specific request.

When changing preference fields or validation rules:

1. Update backend DTOs, views, persistence records, and MikroORM migrations first.
2. Regenerate OpenAPI contracts and generated frontend clients.
3. Update the typed wrappers/hooks in `libs/frontend/api-client/lib/src/auth.ts`.
4. Add tests for valid updates, invalid values, defaults, persistence, compatibility routes, and frontend guest/auth behavior.

## Frontend request, server-state, and shell-state management

The user and admin React apps use TanStack Query for backend request/server state: cacheable reads, mutations, retries, invalidation, and loading/error lifecycles. Do not move server-fetched user/profile/list data into MobX; keep it in TanStack Query so request ownership remains explicit.

MobX is included as the client/UI shell state foundation in `@app/frontend-ui`. The shared `RootStore` owns `LocaleStore`, `AuthShellStore`, and `UiStore` for observable client-only state such as active locale/theme, bearer-token/session shell hints, sidebar/modal state, and future shell state. React local state is still appropriate for small, purely local transient details such as one form field, route snapshots, authenticated preference overrides, or an uncontrolled disclosure.

All frontend backend calls must go through `apiFetch` from `@app/frontend-api-support` (or the compatibility re-export from `@app/frontend-ui`) or the generated `@app/api-client` wrappers that use API support; do not call raw `fetch` from non-test app or shared frontend source outside the API-support implementation. `LocaleStore` keeps the shared API support client synchronized with the current resolved locale, and `apiFetch` injects `Accept-Language` at request time along with JSON and bearer headers. This ensures locale changes affect the next request without duplicating header logic in feature components. API support also localizes its fallback request-failed copy from the current API locale when a problem response does not include localized text.

When adding a backend call:

1. Add or regenerate the typed client wrapper in `@app/api-client` rather than embedding endpoint paths in app code.
2. Use `useQuery` for reads and include stable query-key parts such as domain, resource, token/session scope, and `locale` when localized response data can change.
3. Use `useMutation` for writes, then invalidate affected query-key prefixes with `queryClient.invalidateQueries`.
4. Let `@app/frontend-api-support` own `Accept`, `Accept-Language`, `Content-Type`, bearer header construction, URL resolution, and localized fallback request-failed copy.
5. Add or update tests for headers, query invalidation behavior, generated wrapper use, and the raw-fetch guard.

## Adding a locale or key

Shared translation catalog files live in root `i18n/<locale>/<scope>/<component>.json`; the TypeScript loader and key types live in `libs/common/i18n/lib/src/`.

1. Add the locale code to the supported locale list in `libs/common/i18n/lib/src/locales.ts`.
2. Add every file listed by `localeCatalogFileNames` for the new locale and keep the merged key set in full parity with `i18n/en/**`.
3. Add every new key to every locale catalog and the `TranslationKey` union.
4. Place keys in the matching scoped component file (`landing/app.json`, `admin/users.json`, `common/errors.json`, `user/social-auth.json`, `bots/telegram.json`, and so on). Keep each file at or below 60 keys and 90 non-empty lines; split large domains before exceeding those limits.
5. Keep keys stable and descriptive, for example `landing.title`, `common.language`, `common.theme.dark`, or `errors.bad-request.title`. Landing/user/admin user-facing copy must use keys for visible text, aria labels, placeholders, fallback errors, card/stat labels, title strings, and shared UI defaults. The shared `no-hardcoded-copy` guard catches practical JSX/prop/property regressions.
6. Add tests for locale resolution, interpolation, fallback behavior, catalog key parity, duplicate raw/merged key detection, thin-file policy, and any new domain-specific keys.
7. Update the frontend language switcher labels (`common.language.<locale>`) and any theme-adjacent labels if the preference UI changes.

Missing keys fall back to the default locale before returning the key name, which makes gaps visible during development without crashing production pages.

## Social auth, TMA, bot, and Discord key conventions

Future social-auth and bot surfaces reserve stable key prefixes so runtime work can stay localized from the start:

- `auth.provider.*` for provider labels such as Telegram and Discord.
- `auth.social.*` for social-auth buttons, link/unlink statuses, callback errors, provider conflicts, step-up prompts, last-method guard copy, and create-account flows.
- `tma.*` for Telegram Mini App loading, unsupported-client, authenticated, account-link, and Mini App deep-navigation states.
- `deepNav.*` for provider-neutral deep-link loading, unsupported, authenticated, link-required, expired, and not-found states.
- `bot.menu.*`, `bot.route.*`, `bot.error.*`, and `bot.message.*` for Telegram bot menus, routes, rate-limit/expiry errors, link/unlink errors, and generic bot responses.
- `discord.commands.*`, `discord.components.*`, and `discord.messages.*` for Discord slash-command labels, command descriptions, command responses, component labels, rate-limit copy, auth-required copy, and link-conflict messages.

Keep provider-specific names under the provider prefix (`discord.commands.link.description`) and shared social-auth behavior under `auth.social.*`; store Discord interaction keys in `bots/discord.json` and web social-auth keys in `user/social-auth.json` respectively. Do not put bot tokens, client secrets, webhook secrets, provider ids, or environment-specific callback URLs in locale catalogs. See `docs/social-auth-bots.md` for the planned provider architecture, environment variables, and rollout checklist.

## Frontend usage

Wrap React trees in `FrontendStateProvider` near `FrontendQueryProvider`, then use `FrontendI18nProvider` and `useI18n()` for translations and preference controls:

```tsx
<FrontendStateProvider>
  <FrontendI18nProvider>
    <Page />
  </FrontendI18nProvider>
</FrontendStateProvider>;

const { locale, setLocale, theme, setTheme, t } = useI18n();

setTheme("dark");
return <h1>{t("landing.title")}</h1>;
```

`ProductShell` includes `LanguageSwitcher` and `ThemeSwitcher`, so the landing, user, and admin apps get consistent preference controls. Shared UI tests should pass `initialLocale`/`initialTheme` when they need deterministic copy or DOM attributes.

## Backend exceptions and responses

Backend exceptions use RFC 9457 Problem Details with stable `code`, `type`, `status`, `title`, optional `detail`, and optional `instance` fields. The wire media type remains `application/problem+json`, and repository problem types use stable `urn:problem:*` values. Localization changes only human-readable text (`title` and `detail`); clients should key logic off `code`, `type`, or HTTP status.

Use `BaseException` / `Exception` helpers for application errors:

```ts
throw Exception.forbidden("Required role is missing.");
```

Problem response filters and result mappers resolve the request locale and localize known error codes such as `bad-request`, `forbidden`, `not-found`, `conflict`, `client-data-validation`, `rate-limited`, and `internal-server-error`.

## Validation and rate limits

Validation pipes normalize class-validator failures into localized Problem Details. The problem code remains stable (`client-data-validation`) while field messages can be localized and interpolated. Validation issues are exposed through the `errors[]` extension with per-field `detail` and JSON Pointer `pointer` entries. Rate limit responses use the same problem format and localize the `rate-limited` title/detail.

## Production notes

- Keep error codes stable across releases; changing codes is a client contract change.
- Prefer short backend details that are safe to expose to end users.
- Do not put secrets, tokens, or environment-specific values in translation files.
- Add monitoring around error `code` and `status`, not localized text.
- Include every supported locale and theme preference in smoke tests for critical auth, profile, and admin flows.
