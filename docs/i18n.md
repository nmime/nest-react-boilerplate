# Internationalization (i18n)

The workspace ships with shared English and Spanish translations for frontend copy and backend problem responses. The goal is to keep user-facing text localizable while preserving stable error codes for clients, logs, and alerting.

## Locale detection and persistence order

Backend request locale resolution uses the first supported locale found in this order:

1. `?lang=` or `?locale=` query parameter
2. `x-locale` or `x-language` header
3. `locale` or `lang` cookie
4. `Accept-Language` header
5. Default fallback locale (`en`)

Authenticated frontend sessions additionally persist a profile preference in `auth_users.locale`. The column is nullable for existing and guest users, defaults to no saved preference, and is constrained to supported locale values (`en`/`es`) when present. After a bearer token is available, the user and admin frontends load `/auth/me` and/or profile data, apply the saved user locale first, then continue using the standard browser-safe fallback order if no saved preference exists:

1. saved authenticated user locale from `/auth/me` or profile data
2. `?lang=` or `?locale=` query parameter
3. `boilerplate.locale` in `localStorage`
4. `locale` or `lang` cookie
5. `navigator.languages` / `navigator.language`
6. Default fallback locale (`en`)

Guests use only the browser-safe fallback order starting at the URL/localStorage/cookie/browser/default steps.

The frontend language switcher always persists the selected locale locally in `localStorage` and a `locale` cookie. For authenticated users it also calls `PATCH /auth/me/locale` so the preference follows future sessions and is included in auth views/JWT principals. User and admin frontend API calls must always send `Accept-Language` with the current resolved locale; backend APIs still localize each request from that frontend-provided header (or the backend fallback chain above).

## Frontend request, server-state, and shell-state management

The user and admin React apps use TanStack Query for backend request/server state: cacheable reads, mutations, retries, invalidation, and loading/error lifecycles. Do not move server-fetched user/profile/list data into MobX; keep it in TanStack Query so request ownership remains explicit.

MobX is included as the client/UI shell state foundation in `@app/frontend-ui`. The shared `RootStore` owns `LocaleStore`, `AuthShellStore`, and `UiStore` for observable client-only state such as the active locale, bearer-token/session shell hints, theme/sidebar/modal state, and future domain shell state. React local state is still appropriate for small, purely local transient details such as one form field, route snapshots, authenticated-locale overrides, or an uncontrolled disclosure.

All frontend backend calls must go through `apiFetch` from `@app/frontend-ui`; do not call raw `fetch` from non-test app or shared frontend source outside the API client implementation. `LocaleStore` keeps the shared API client synchronized with the current resolved locale, and `apiFetch` injects `Accept-Language` at request time along with JSON and bearer headers. This ensures locale changes affect the next request without duplicating header logic in feature components. The API client also localizes its fallback request-failed copy from the current API locale when a problem response does not include localized text.

When adding a backend call:

1. Add a small typed fetcher that calls `apiFetch<T>(path, { baseUrl, authToken, json, method })`.
2. Use `useQuery` for reads and include stable query-key parts such as domain, resource, token/session scope, and `locale` when localized response data can change.
3. Use `useMutation` for writes, then invalidate affected query-key prefixes with `queryClient.invalidateQueries`.
4. Let `apiFetch` own `Accept`, `Accept-Language`, `Content-Type`, and bearer header construction.
5. Add or update tests for headers, query invalidation behavior, and the raw-fetch guard.

## Adding a locale or key

Shared translation resources live in `libs/common/i18n/src/`.

1. Add the locale code to the supported locale list.
2. Add a translation file or object for every key used by the apps and APIs.
3. Keep keys stable and descriptive, for example `landing.title`, `common.language`, or `errors.bad-request.title`. Landing/user/admin user-facing copy must use keys for visible text, aria labels, placeholders, fallback errors, card/stat labels, title strings, and shared UI defaults. The shared `no-hardcoded-copy` guard catches practical JSX/prop/property regressions.
4. Add tests for locale resolution, interpolation, fallback behavior, and any new domain-specific keys.
5. Update the frontend language switcher labels (`common.language.<locale>`).

Missing keys fall back to the default locale before returning the key name, which makes gaps visible during development without crashing production pages.

## Frontend usage

Wrap React trees in `FrontendStateProvider` near `FrontendQueryProvider`, then use `FrontendI18nProvider` and `useI18n()` for translations:

```tsx
<FrontendStateProvider>
  <FrontendI18nProvider>
    <Page />
  </FrontendI18nProvider>
</FrontendStateProvider>;

const { locale, setLocale, t } = useI18n();

return <h1>{t("landing.title")}</h1>;
```

`ProductShell` includes `LanguageSwitcher`, so the landing, user, and admin apps get consistent language controls. Shared UI tests should pass an `initialLocale` when they need deterministic copy.

## Backend exceptions and responses

Backend exceptions use RFC 7807 problem details with stable `code`, `type`, `status`, `title`, and optional `detail` fields. Localization changes only human-readable text (`title` and `detail`); clients should key logic off `code` or HTTP status.

Use `BaseException` / `Exception` helpers for application errors:

```ts
throw Exception.forbidden("Required role is missing.");
```

Problem response filters and result mappers resolve the request locale and localize known error codes such as `bad-request`, `forbidden`, `not-found`, `conflict`, `validation-error`, `too-many-requests`, and `internal-server-error`.

## Validation and rate limits

Validation pipes normalize class-validator failures into localized problem details. The problem code remains stable (`validation-error`) while field messages can be localized and interpolated. Rate limit responses use the same problem format and localize the `too-many-requests` title/detail.

## Production notes

- Keep error codes stable across releases; changing codes is a client contract change.
- Prefer short backend details that are safe to expose to end users.
- Do not put secrets, tokens, or environment-specific values in translation files.
- Add monitoring around error `code` and `status`, not localized text.
- Include every supported locale in smoke tests for critical auth, profile, and admin flows.
